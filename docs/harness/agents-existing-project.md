# Старый проект

Чужой, старый или свой код без агентов. Код пишет ИИ.

## Сначала: скопируйте болванку

Скопируйте в папку старого проекта:
- `.claude/` *(включая `.claude/agents/`)*
- `CLAUDE.md`
- `docs/harness/` *(по желанию)*

Откройте терминал в **папке проекта** и запустите `claude` — агенты и старый код в одном месте.

## Цикл

| # | Вы | Что происходит |
|---|-----|----------------|
| 1 | `@code-explorer объясни простыми словами, что делает проект` | Понимаете структуру |
| 2 | `@planner хочу добавить ...` | План |
| 3 | Обсудить план **или** `Реализуй по плану` | Уточняете **или** ИИ пишет код |
| 4 | `@code-reviewer проверь` | Проверка |
| 5 | `Исправь критичное` | Правки |

Если **не запускается** — сначала `@build-error-resolver почини запуск`, потом planner.

**Когда code-explorer можно пропустить:** правка мелкая (опечатка, текст, одна кнопка) или вы уже разбирали этот проект в этом же чате — сразу `@planner хочу ...`. **Когда нужен:** первый заход в проект, задача затрагивает несколько файлов, или непонятно как всё устроено — без explorer план рискует не учесть существующие паттерны.

**Planner сохраняет план в `docs/plans/<дата>-<название>.md`** со статусом фаз (⬜/🔄/✅). Сессия прервалась — новый чат → `Продолжи по docs/plans/<файл>`. См. [docs/plans/README.md](../plans/README.md).

На шаге 3 (реализация) TDD включается автоматически — упоминать его не нужно, это часть пайплайна.

## Planner или code-architect?

| | Planner | Code-architect |
|--|---------|----------------|
| **Когда** | Почти всегда | Большая фича в **уже большом** коде |
| **Пример** | добавить кнопку, форму | встроить поиск в огромный каталог |

**Обычно:** explorer → planner — достаточно.

**Code-architect** — **до** planner, если фича сложная и код большой:

```
1. @code-explorer как устроен каталог
2. @code-architect как встроить поиск по 10 000 товаров
3. @planner план по этой схеме
4. Реализуй по плану
5. @code-reviewer проверь
```

## Пример: плохая архитектура — хочу переделать

Когда проект работает, но устроен плохо: всё в одном файле, нет структуры, сложно добавлять новое.

```
1. @code-explorer объясни архитектуру проекта — что устроено плохо и почему
2. @architect предложи лучшую архитектуру на основе того, что нашёл explorer
3. @planner составь план перехода с текущей архитектуры на новую, по шагам
4. (опционально, если есть важная бизнес-логика) @tdd-guide напиши тесты на текущее поведение — страховка перед переносом
5. Реализуй шаг 1 из плана
6. @code-reviewer проверь
7. Исправь критичное — и так по каждому шагу
```

**Важно:** переделывать архитектуру лучше маленькими шагами, а не всё сразу — меньше риск сломать работающее.

**Когда нужен шаг с tdd-guide:** код двигает деньги, считает что-то важное или его поведение легко сломать незаметно. **Когда не нужен:** просто переносите файлы/классы по папкам без изменения логики.

## Пример: добавить поиск

```
1. @code-explorer как сейчас устроен каталог
2. @planner хочу добавить поиск по названию
3. Обсудите план — или сразу: «Реализуй по плану»
4. @code-reviewer проверь
5. Исправь критичное
```

## Пример: первый заход в проект

```
1. @code-explorer что делает этот проект, простыми словами
2. @build-error-resolver не запускается   ← только если не работает
3. @planner хочу обновить главную страницу
4. Обсудите план — или сразу: «Реализуй по плану»
5. @code-reviewer проверь
```

## Когда звать ещё кого-то

| Ситуация | Агент | Пример |
|----------|-------|--------|
| TypeScript / JS файлы | typescript-reviewer | `@typescript-reviewer проверь .ts файлы` |
| Python-проект | python-reviewer | `@python-reviewer проверь .py файлы` |
| Формы, данные, API | security-reviewer | `@security-reviewer проверь форму` |
| Тихая ошибка | silent-failure-hunter | `@silent-failure-hunter молча не сохраняет` |
| Много мусора в коде | refactor-cleaner | `@refactor-cleaner убери лишнее` |
| Код раздулся | code-simplifier | `@code-simplifier упрости, ничего не ломая` |
| Есть Qt/Tkinter GUI, нужна веб-версия | qt-html-mirror-builder | `@qt-html-mirror-builder сделай HTML версию GUI` |
| Нет README | docs-writer | `@docs-writer как запустить проект` |
| Документация устарела после правок | doc-updater | `@doc-updater обнови карту проекта` |

**docs-writer vs doc-updater:** docs-writer пишет README/гайды с нуля для людей. doc-updater обновляет уже существующую документацию и карту файлов проекта после правок в коде.

## Если сломалось после правок

```
«После обновления не открывается вход»
→ ИИ чинит в чате
→ @code-reviewer проверь фикс
```

## Не знаешь кого звать?

```
@advisor [опиши что происходит в проекте]
```

## Правила

1. Сначала **понять** (explorer), потом **менять** (planner).
2. Маленькими шагами — меньше ломается.
3. Код не пишете вы — только задачи словами.
